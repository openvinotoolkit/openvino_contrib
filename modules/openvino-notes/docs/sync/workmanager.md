# Модуль синхронизации: Фоновые задачи (WorkManager)

Для обеспечения гарантированной доставки изменений на сервер даже в случае закрытия приложения пользователем, вся инфраструктура запуска синхронизации вынесена из главного процесса UI в системный планировщик **Jetpack WorkManager**.

## 1. Планирование задач через WorkManagerSyncScheduler

Запуск фонового процесса инкапсулирован в классе `WorkManagerSyncScheduler`, реализующем доменный интерфейс `SyncScheduler`. Вместо хаотичного вызова корутин из слоев интерфейса (`ViewModel`), отправка задачи триггерится системными намерениями.

### Критерии, ограничения (Constraints) и Backoff
Для минимизации расхода заряда батареи и исключения ошибок холостых сетевых запросов настраиваются жесткие системные ограничения и политика повторов:

```kotlin
val constraints = Constraints.Builder()
    .setRequiredNetworkType(NetworkType.CONNECTED) // Обязательное наличие интернета
    .build()

val syncRequest = OneTimeWorkRequestBuilder<SyncWorker>()
    .setConstraints(constraints)
    .setInputData(workDataOf("USER_ID" to userId))
    .setBackoffCriteria(
        BackoffPolicy.EXPONENTIAL, // Экспоненциальное увеличение задержки при ошибках
        10,
        TimeUnit.SECONDS,
    ).build()
```

### Обеспечение уникальности выполнения
Синхронизация тяжелых файлов и JSON-структур требовательна к транзакционности. Для предотвращения состояния гонки (Race Condition), когда несколько триггеров пытаются одновременно писать данные в одну и ту же учетную запись Firebase, используется механизм уникальных воркеров:

```kotlin
workManager.enqueueUniqueWork(
    "sync_work_$userId",            // Уникальное имя воркера, привязанное к UID пользователя
    ExistingWorkPolicy.REPLACE,     // Стратегия: если прошлый воркер еще работает, заменить его новым
    syncRequest
)
```

Использование стратегии ExistingWorkPolicy.REPLACE гарантирует, что если пользователь совершил серию быстрых изменений в заметках, цепочка промежуточных воркеров прервется и запустится один актуальный процесс, содержащий последний слепок локальной БД.

## 2. Реализация SyncWorker и валидация сессии
Класс SyncWorker является точкой входа, активируемой операционной системой Android. Он наследуется от CoroutineWorker, предоставляя встроенную поддержку асинхронных операций Kotlin.

### 2.1 Порядок извлечения контекста и токена:

Воркер извлекает USER_ID из переданного inputData. Если его нет, запрашивает у AuthManager.getCurrentUserId(). Если пользователь не авторизован, задача моментально завершается с Result.failure().

### 2.2 Критический шаг — обновление токена:
 Воркер вызывает authManager.refreshAuthToken(). Поскольку Firebase-токены имеют срок жизни, фоновый процесс может запуститься с "протухшим" токеном и получить ошибку 401. Если токен временно невозможно обновить (например, упали сервера авторизации), воркер откладывает работу, возвращая Result.retry().

## 3. Политика обработки ошибок и перезапусков (Retry Policy)
Внутри метода doWork() весь блок синхронизации обернут в защитный контейнер обработки исключений. В зависимости от природы возникшей ошибки SyncWorker дифференцирует возвращаемый операционной системе результат:

| Тип исключения / Сценарий | Возвращаемый результат | Поведение системы |
| :--- | :--- | :--- |
| **Успешное выполнение** | `Result.success()` | Задача завершается. Локальные данные полностью консистентны с облаком Firebase. |
| **CancellationException** | `throw e` | Сигнализирует о том, что корутина была отменена изнутри ОС (например, сработала политика REPLACE). Перевыбрасывается для корректной остановки воркера. |
| **Невалидный токен / Сбой авторизации** | `Result.retry()` | Firebase-токен недоступен. Запуск откладывается до восстановления связи. |
| **IOException / FirebaseException** | `Result.retry()` | Повтор выполнения. Сбой сети, тайм-аут сокета или падение серверов Firebase Storage. WorkManager ставит задачу в очередь с экспоненциальной задержкой. |
| **Иные исключения (в т.ч. SerializationException)** | `Result.failure()` | Критический сбой. Ошибка парсинга JSON, битая структура данных или неизвестная ошибка логики. Задача прерывается без права на автоматический повтор, чтобы избежать бесконечного разряда батареи. |


```kotlin
return try {
    syncManager.sync(userId)
    Result.success()
} catch (e: CancellationException) {
    throw e
} catch (e: java.io.IOException) {
    Result.retry() // Сетевая ошибка, повторить позже
} catch (e: com.google.firebase.FirebaseException) {
    Result.retry() // Ошибка облака, повторить позже
} catch (e: Exception) {
    Result.failure() // Критическая ошибка логики, остановить выполнение
}
```
