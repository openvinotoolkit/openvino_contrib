# OpenVINO GFX Plugin (Experimental)

## 1. Overview
- Experimental Metal GPU backend for OpenVINO, built on MLIR → autogenerated **MSL** kernels.
- Target platform: Apple Silicon / macOS with Metal enabled.
- Status: work‑in‑progress; executes multi‑op graphs via MLIR → MSL pipeline with segment dispatch.

## 2. Supported OpenVINO version
- The plugin currently targets **OpenVINO 2026.0.0** (Developer Package).
- It is intended to be built against an OpenVINO Developer Package produced from a build tree, e.g.
  `<openvino-build>/install/cmake` (replace with your local path).

## 3. Architecture (short)
- **Plugin (`src/plugin/plugin.cpp`)** implements `ov::IPlugin`, registers `"GFX"`, exposes device properties and `query_model`.
- **CompiledModel / InferRequest (`src/plugin/compiled_model.*`, `src/plugin/infer_request.*`)**
  - `CompiledModel` keeps IR of lowered ops plus compiled MSL pipelines.
  - `InferRequest` allocates public `ov::Tensor` inputs/outputs, wraps them, and dispatches to the runtime executor.
- **MLIR backend (`src/runtime/mlir_backend.mm`)**
  - Lowers supported graphs to MLIR, then to autogenerated Metal Shading Language (MSL) kernels.
- Fallback path exists (CPU) for unsupported ops; execution uses MLIR → MSL only.

## 4. Execution really runs on GPU
- Metal device/command queue are created via `MTLCreateSystemDefaultDevice()` and used to run compiled MSL pipelines.
- Outputs are copied back to host only at the end of execution.

**How to observe GPU load**
```bash
# Terminal 1: watch GPU power/load
sudo powermetrics --samplers gpu_power -i 500 -n 10

# Terminal 2: run the GFX tests
DYLD_LIBRARY_PATH=<path-to-openvino-runtime-libs> \
  <build-dir>/modules/gfx_plugin/tests/ov_gfx_func_tests --gtest_filter=MetalBasicOps.*
```
You should see GPU power/load spikes while the tests are running.

## 5. Supported ops (summary)
The MLIR → MSL backend covers the common compute path and a growing set of layout/shape ops. For the
full list and per‑op constraints, see `GFX-backend-state.md`.

Summary (non‑exhaustive):
- MatMul (rank‑2/3/4, batch variants)
- Convolution 2D/3D, GroupConv (static layout constraints)
- Eltwise Add/Sub/Mul/Div/Pow/Mod/FloorMod (broadcast with constraints)
- Unary activations: Relu/Sigmoid/Tanh/Elu/PRelu/Gelu/Swish/Abs/Sign/Identity
- Pooling: MaxPool / AvgPool
- Softmax
- BatchNormInference
- Concat / Split / Slice / Transpose / Reshape / Convert / Interpolate

**Known limitations (high‑level)**
- Dynamic shapes are limited; many ops require static ranks/params.
- Partial offload to CPU is disabled (unsupported ops/params throw).
- Device selection/multi‑device and remote tensors are not fully implemented.
- Dtypes beyond f16/f32/i32/i64/u8 are limited.

## 6. Building the module
Assuming OpenVINO 2026 DevPackage is already built and installed to `<openvino-build>/install/cmake`:
```bash
# 1) (example) build OpenVINO DevPackage
cd /path/to/openvino
cmake -S . -B build-ninja -G Ninja -DENABLE_TESTS=OFF
cmake --build build-ninja --target install

# 2) configure gfx_plugin
cd /path/to/openvino_contrib
mkdir -p cmake-build-metal && cd cmake-build-metal
cmake -S ../modules/gfx_plugin -B . \
  -DCMAKE_BUILD_TYPE=Release \
  -DOpenVINODeveloperPackage_DIR=/path/to/openvino/build-ninja/install/cmake \
  -DENABLE_TESTS=ON

# 3) build plugin + tests
cmake --build . --target ov_gfx_func_tests -- -j"$(sysctl -n hw.ncpu 2>/dev/null || echo 4)"
```
Adjust paths to match your local build directories.

## 7. Running tests
Available gtest suite: `ov_gfx_func_tests`, covering:
- MetalBasicOps.* (core ops, device properties, AUTO/HETERO sanity)
- MetalPrecisionStudy.* (diagnostic accuracy checks)
- Behavior/core_config/remote/executable_network tests (plugin API coverage)

Run all:
```bash
DYLD_LIBRARY_PATH=<path-to-openvino-runtime-libs> \
  ./modules/gfx_plugin/tests/ov_gfx_func_tests --gtest_filter=MetalBasicOps.*
```
If the GFX plugin cannot be registered (e.g., missing Metal device), tests will gracefully `GTEST_SKIP`.

## 8. Current limitations / roadmap
- Apple Metal GPU only (Apple Silicon / macOS).
- Op coverage is growing but constrained by static‑shape/param requirements.
- Performance tuning is minimal; `ov::hint::performance_mode` is stored but not yet exploited.
- TODO / roadmap:
  - Expand dynamic shape handling and relax const‑param constraints.
  - Enable partial offload / graceful fallback where feasible.
  - Add device selection and real RemoteContext/RemoteTensor support.
  - Extend dtype coverage (BF16/INT8) and add more fusions/optimizations.
